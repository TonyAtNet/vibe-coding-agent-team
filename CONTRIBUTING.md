## 贡献方式

### 你可以贡献什么

- **新增角色**：发现 Vibe Coding 链路中有缺失的角色？提交一个新的 `.md` 文件
- **改进现有角色**：优化 Prompt、更新工具链、修正错误链接、补充成功指标
- **修复问题**：报告并修复角色内容中的事实错误、过时工具链、失效链接
- **补充示例**：在 `examples/` 中添加多 Agent 协作的实战案例
- **翻译**：将角色文件翻译成其他语言（英文优先）
- **文档改进**：优化 README、使用指南、站点体验

### 暂不接受的内容

- 纯营销性质的内容
- 与 Vibe Coding 无关的传统软件开发角色
- 包含 emoji 或角色扮演的文件

## 角色设计规范

每个新角色必须是一个 `.md` 文件，存放在 `product/` 或 `engineering/` 目录下，文件名格式为 `vibe-xxx.md` 或 `product-manager.md`（已有范例）。

### 文件结构

```markdown
---
name: vibe-role-name
description: 一句话描述，50 字以内
color: color-name  # 可选：red, blue, green, yellow, magenta, cyan, indigo, orange, teal, purple
---

# 角色中文名

本智能体专为 Vibe Coding 与 AI-Native 产品流程构建，...

可操作的现代工具链覆盖：
- 分类：tool1, tool2, tool3
...

---

## 核心使命

（1-2 段，说明角色的核心目标和产出物）

核心产出：
- 产出 1
- 产出 2
...

---

## 关键原则

1. 原则一（解释为什么）
2. 原则二（解释为什么）
...

---

## 技术交付物

### 交付物名称

```markdown
模板内容
```

---

## 工作流程

### 第一步：XXX

- 步骤 1
- 步骤 2
...

### 第二步：XXX

...

---

## 成功指标

- 指标 1：具体数值目标
- 指标 2：具体数值目标
...

---

## 沟通风格

- 风格 1：示例对话
- 风格 2：示例对话
...
```

### 必备章节

| 章节 | 要求 | 说明 |
|------|------|------|
| YAML Frontmatter | 必须 | 包含 `name`, `description`, `color` |
| 核心使命 | 必须 | 角色定位 + 核心产出清单 |
| 关键原则 | 必须 | 3-8 条，每条有解释 |
| 技术交付物 | 必须 | 至少 1 个可执行的 Spec 模板 |
| 工作流程 | 必须 | 分步骤，每步有操作清单 |
| 成功指标 | 必须 | 至少 5 个可量化的指标 |
| 沟通风格 | 必须 | 至少 3 个风格示例 |
| AI 工具链 | 必须 | 列出可操作的现代工具 |
| AI 可观测性 | 建议 | 幻觉率、TTFT、Token 成本、人工接管率 |
| Vibe Speed / RICE-V | 建议 | 产品/优先级相关角色必备 |

### 禁止项

- **Emoji**：所有文件内容不得出现 emoji（包括章节图标、列表符号等）
- **角色扮演**：不得出现 "你是 Alex"、"10 年经验专家"、"资深工程师" 等虚假人格设定。角色应以 "本智能体" 自称
- **静态文档心态**：交付物必须是 "可执行 Spec"（包含 Prompt 模板、代码配置、检查清单），而非纯概念描述
- **过时工具链**：工具链应聚焦 2025-2026 年的 AI-Native 工具（Cursor, v0, Lovable, MCP, LangChain 等），避免大量传统工具
- **空泛指标**：避免 "提升用户体验" 等不可量化指标，必须给出具体数值

## 提交流程

### 1. Fork 仓库

点击 GitHub 页面右上角的 "Fork" 按钮，将仓库复制到你的账户下。

### 2. 创建分支

```bash
git checkout -b feature/vibe-new-role-name
# 或
git checkout -b fix/update-vibe-prototyper-tools
```

分支命名规范：
- 新增角色：`feature/vibe-role-name`
- 修复问题：`fix/short-description`
- 文档改进：`docs/short-description`

### 3. 修改并提交

```bash
git add .
git commit -m "Add vibe-new-role: AI-Native XXX 工程师"
# 或
git commit -m "Fix: 更新 vibe-prototyper 工具链，替换 Bolt 为 Tempo"
```

提交信息规范：
- 新增角色：`Add vibe-xxx: [一句话描述]`
- 修复问题：`Fix: [具体修复内容]`
- 更新工具链：`Update: [角色名] 工具链`

### 4. 本地验证（重要）

在提交前，请运行验证脚本：

```bash
python scripts/validate.py
```

如果验证失败，请修复后再提交。详见 [本地验证](#本地验证)。

### 5. 推送并创建 PR

```bash
git push origin feature/vibe-new-role-name
```

然后到 GitHub 上创建 Pull Request，选择模板填写。

## 检查清单

提交前，请对照以下清单自检：

### 新角色检查清单

- [ ] 文件存放在正确的目录（`product/` 或 `engineering/`）
- [ ] 文件名格式正确（`vibe-xxx.md` 或已有范例格式）
- [ ] 包含 YAML Frontmatter，且包含 `name`, `description`, `color`
- [ ] 不含 emoji（可用 `scripts/validate.py` 检查）
- [ ] 不含虚假角色扮演（"你是 Alex"、"10 年经验" 等）
- [ ] 包含所有必备章节（核心使命、关键原则、技术交付物、工作流程、成功指标、沟通风格）
- [ ] 技术交付物包含至少 1 个可执行的 Spec 模板（含代码块）
- [ ] 成功指标可量化（有具体数值，非空泛描述）
- [ ] 列出可操作的工具链（每个工具都应是当前可用的）
- [ ] 包含 AI 可观测性指标（建议覆盖幻觉率、TTFT、Token 成本，产品/优先级角色可豁免）
- [ ] 本地验证通过（`python scripts/validate.py` 无错误）

### 修改现有角色检查清单

- [ ] 说明修改原因（在 PR 描述中）
- [ ] 修改后仍符合所有禁止项要求
- [ ] 本地验证通过

## 本地验证

我们提供了自动验证脚本，帮助你检查提交内容是否符合规范。

### 安装依赖

无需额外依赖，仅使用 Python 标准库。

### 运行验证

```bash
python scripts/validate.py
```

### 验证内容

脚本会自动检查以下项目：

1. **Frontmatter 完整性**：每个 `.md` 文件必须包含 `name`, `description`, `color`
2. **Emoji 检测**：文件内容不得包含 emoji（U+1F300 及以上 Unicode 字符）
3. **角色扮演检测**：不得出现 "你是"、"资深"、"专家"、"10 年经验" 等虚假人格关键词
4. **必备章节检测**：必须包含核心使命、关键原则、技术交付物、工作流程、成功指标、沟通风格
5. **工具链检测**：必须包含 "可操作的现代工具链" 或 "工具链" 相关章节
6. **AI 可观测性检测**：建议包含 "幻觉率"、"TTFT"、"Token" 等关键词。仅输出 WARN 提示，不影响通过

### 验证失败怎么办

验证脚本会输出详细的错误信息，包括：
- 哪个文件出了问题
- 具体违反了哪条规则
- 建议的修复方向

请根据错误信息修复后重新运行验证。

## 评审流程

提交 PR 后，维护者会进行以下检查：

1. **自动化检查**：GitHub Actions 运行 `validate.py`，必须全部通过
2. **内容评审**：维护者检查角色设计的合理性、工具链的时效性、指标的可量化性
3. **格式评审**：检查是否符合文件结构和命名规范

通常会在 3-5 个工作日内回复。如需修改，请在 PR 中继续提交更改。
